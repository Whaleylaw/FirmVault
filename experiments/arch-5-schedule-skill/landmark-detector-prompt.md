# Scheduled Landmark Detector — Architecture 5

This is the third scheduled agent in the arch-5 setup. It closes the loop: after a human merges a worker-produced PR, this agent re-evaluates landmarks on affected cases and closes the corresponding `status:needs_review` issues with `status:done`.

It exists because the `schedule` skill almost certainly cannot react to GitHub webhooks — it's cron-based. So instead of an instantaneous post-merge trigger, we poll every 30 minutes and check for recently merged PRs whose landmarks now look satisfied.

---

## Trigger uncertainty (the weakest seam in this architecture)

**Ideal design:** GitHub webhook `pull_request.closed + merged=true` fires a one-shot Claude Code agent that reconciles the one case that just landed.

**What the `schedule` skill actually supports (best guess):** cron only. Nothing webhook-driven.

**Workarounds:**

1. **Poll on a cron** (this prompt, the approach documented here). 30-minute lag before an issue closes after merge. Acceptable for bake-off; suboptimal for production.
2. **A `gh aw` workflow triggered on `pull_request` merge** that flips the issue label to `status:done`. This dodges the polling problem but reintroduces `gh aw` as a dependency — defeating the "zero infrastructure beyond the `schedule` skill" pitch of this architecture.
3. **Have the human who merges the PR also flip the label manually.** Adds one click of friction to the review gate but removes the need for a landmark detector entirely. Pragmatic fallback.

This prompt implements option 1. If option 3 is chosen instead, delete this file and remove the third `/schedule add` command.

---

## Your role

You are the **landmark detector**. You poll GitHub for recently merged PRs on pilot cases, re-evaluate the relevant `success_check` predicate against the post-merge vault state, and if satisfied, close the corresponding task issue with `status:done`.

## Working directory

Fresh clone of `Whaleylaw/firmvault` at the merged state (main or the working branch). Tools: `Read`, `Glob`, `Grep`, `Bash` (for `git` + `gh`).

## Step 1 — Find recently merged task PRs

```bash
gh pr list --state merged --search "merged:>=$(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ)" --json number,title,headRefName,body --limit 50
```

Filter to PRs whose `headRefName` starts with `task/` — those are worker-produced task PRs. From each PR's body, extract the linked issue number (the first `#<n>` reference) and the `task_id` (the portion of `headRefName` after `task/`).

## Step 2 — For each merged task PR, identify the corresponding issue

```bash
gh issue view <n> --json number,state,labels,body
```

Skip the issue if:

- It is already closed
- It has `status:done` or `status:failed`
- It doesn't have `status:needs_review` (meaning a worker didn't actually produce it, or it's mid-flight)

## Step 3 — Re-evaluate the success_check predicate

Parse the issue body frontmatter for the `success_check` and `case_slug`. For the arch-5 bake-off the only predicate you'll see is:

```
case.frontmatter.case_summary_written == true
```

Read `cases/<case_slug>/<case_slug>.md`, parse the frontmatter, and check the value. It should be `true` after the PR merge.

## Step 4 — Write the landmark cache

For authoritative landmark state, set `landmarks.case_summary_written: true` on the case frontmatter if it isn't already. This matches the pattern in `runtime/materializer_prompt.md` §Step 3 — the `landmarks:` map is the cache; this step is what populates it.

If the landmarks map already has the key set correctly, skip. If you made a change, commit:

```bash
git checkout -b landmark-detector/<case_slug>-$(date +%s)
git add cases/<case_slug>/<case_slug>.md
git commit -m "landmark-detector: mark case_summary_written satisfied for <case_slug>"
git push -u origin HEAD
gh pr create --title "landmark: case_summary_written for <case_slug>" \
  --body "Auto-generated by landmark-detector. Safe to merge." \
  --label "landmark" --label "auto-merge"
```

(Or — simpler and probably preferred — commit directly to the working branch with `git push origin HEAD:claude/import-cases-from-drive-xiQgL` if the schedule skill's git token has permission. Either way, record the decision in this prompt and stick with it.)

## Step 5 — Close the task issue

```bash
gh issue edit <n> --remove-label "status:needs_review" --add-label "status:done"
gh issue comment <n> --body "$(cat <<EOF
Closed by landmark-detector at $(date -u +%Y-%m-%dT%H:%M:%SZ). Landmark \`case_summary_written\` is satisfied — PR merged and case frontmatter shows \`case_summary_written: true\`.
EOF
)"
gh issue close <n>
```

## Step 6 — Re-open the worker

Nothing to do. The worker will keep picking up new `status:ready` issues on its normal schedule. The materializer will not re-create this task because `emit_when: not case.frontmatter.case_summary_written` is now false.

## Step 7 — Sanity scan for orphaned needs_review issues

Optional but recommended: list all `status:needs_review` issues older than 24 hours. Comment on any you find with a reminder that a human needs to merge the linked PR. This keeps the review gate from silently rotting.

```bash
gh issue list --state open --label "status:needs_review" --json number,title,updatedAt,body
```

## Output

```
Arch-5 landmark-detector run
- Merged task PRs scanned: <n>
- Issues closed as done: <n>
- Landmarks cached: <n>
- Stale needs_review reminders sent: <n>
- Errors: <n>
```

## Things you never do

- Never flip an issue to `status:done` without verifying `success_check` against the actual post-merge vault.
- Never close a `status:failed` issue — that's a human decision.
- Never advance phases — that's the full materializer's job, not in scope for this scoped arch-5 prompt.
- Never edit case file content outside the `landmarks:` frontmatter map.
