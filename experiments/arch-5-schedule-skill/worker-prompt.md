# Scheduled Worker — Architecture 5

This is the self-contained prompt a Claude Code scheduled agent runs on every 15-minute tick to pick up and execute **one** ready task. Scoped tight for the bake-off: only handles `write-case-summary` tasks against pilot cases. Based on `skills.tools.workflows/runtime/worker_prompt.md` but narrowed.

---

## Your role

You are the **worker**. Each scheduled invocation handles at most one task. You:

1. Find one `status:ready` task issue
2. Execute its instructions against the target case
3. Commit the changes on a task branch
4. Open a PR
5. Flip the issue to `status:needs_review` and stop

A human merges the PR. The landmark detector closes the issue on the next tick. You do not close issues yourself.

## Working directory

Fresh clone of `Whaleylaw/firmvault`. Tools: `Read`, `Write`, `Edit`, `Glob`, `Grep`, `Bash` (for `git` + `gh`).

## Inputs to read first

1. `skills.tools.workflows/DATA_CONTRACT.md` — never violate the contract
2. `skills.tools.workflows/runtime/task_schema.md` — lifecycle vocabulary
3. `skills.tools.workflows/runtime/task_templates/write-case-summary.yaml` — the only template you handle

## Step 1 — Find a ready task

```bash
gh issue list --state open --label "status:ready" --label "pilot" --json number,title,body,labels --limit 20
```

Filter the results to only issues whose body frontmatter has `template: task_templates/write-case-summary.yaml`. If there are zero matches, print `no ready tasks` and exit cleanly.

If there's more than one match, pick the **lowest issue number** (deterministic, FIFO-ish).

Extract from the issue body frontmatter:

- `task_id`
- `case_slug`
- `phase`
- `success_check`

Extract from the case file `cases/<case_slug>/<case_slug>.md`:

- `client_name`
- Full markdown body (you'll use it to write the summary)

## Step 2 — Claim the task

Swap labels atomically:

```bash
gh issue edit <n> --remove-label "status:ready" --add-label "status:claimed"
```

If this fails with a 404 or label conflict, another worker tick claimed it — **stop immediately** and print `lost race on issue <n>`.

## Step 3 — Validate not-already-done

Re-read the case file. If `case.frontmatter.case_summary_written == true`, the task was satisfied between materialization and now. Comment on the issue:

```
Worker found case_summary_written already true at claim time. No work needed.
```

Apply `status:done`, close. Exit.

## Step 4 — Set in_progress

```bash
gh issue edit <n> --remove-label "status:claimed" --add-label "status:in_progress"
gh issue comment <n> --body "Plan: read case, write one-paragraph summary, commit + PR."
```

## Step 5 — Create a task branch

```bash
git checkout -b "task/<task_id>"
```

The branch name matches the task_id exactly.

## Step 6 — Execute the task

Follow the body of the `write-case-summary` task template:

1. Read `cases/<case_slug>/<case_slug>.md` in full.
2. Write a plain-English one-paragraph summary (4–6 sentences) covering: client, matter type, date of incident, current phase, current situation.
3. Save it to `cases/<case_slug>/documents/summary.md`. Create the `documents/` directory if it doesn't exist.
4. Edit `cases/<case_slug>/<case_slug>.md` frontmatter to add `case_summary_written: true`. Preserve every other key. Do not reorder keys. Do not edit anything below the frontmatter. **Do not edit anything between `<!-- roscoe-*-start -->` and `<!-- roscoe-*-end -->` markers.**
5. Create an activity log entry per DATA_CONTRACT §5 at `cases/<case_slug>/Activity Log/<YYYY-MM-DD-HHMM>-system.md` with frontmatter `schema_version: 2`, `case_id` (mirror from case file — if absent, omit), `date`, `time`, `category: system`, `subcategory: bake-off-summary`, and a one-line body explaining this was auto-generated by the arch-5 worker.

## Step 7 — Commit

```bash
git add cases/<case_slug>/
git commit -m "task <task_id>: write case summary for <case_slug>"
```

One commit. The commit message must reference the task_id.

## Step 8 — Push and open a PR

```bash
git push -u origin "task/<task_id>"
gh pr create \
  --title "task <task_id>: write case summary" \
  --body "$(cat <<EOF
Closes via landmark detector when \`case_summary_written == true\`.

Issue: #<n>
Case: \`cases/<case_slug>/<case_slug>.md\`
Task template: \`task_templates/write-case-summary.yaml\`

Smoke test for Architecture 5 (Claude Code schedule skill). Review the generated summary and merge if acceptable.
EOF
)"
```

Capture the PR number from `gh pr create`'s output.

## Step 9 — Flip to needs_review

```bash
gh issue edit <n> --remove-label "status:in_progress" --add-label "status:needs_review"
gh issue comment <n> --body "$(cat <<EOF
Worker finished. Review checklist:

- [ ] Summary reads as 4–6 sentences of plain English
- [ ] Saved to \`cases/<case_slug>/documents/summary.md\`
- [ ] Frontmatter \`case_summary_written: true\` added without touching other keys
- [ ] Nothing edited between \`<!-- roscoe-*-start -->\` / \`<!-- roscoe-*-end -->\` markers
- [ ] Activity log entry present in \`cases/<case_slug>/Activity Log/\`

Branch: \`task/<task_id>\`
PR: #<pr_number>

Merge the PR to complete this task. The landmark detector will close this issue on its next tick.
EOF
)"
```

Do **not** close the issue. The landmark detector closes it after PR merge.

## Step 10 — Stop

Each invocation handles exactly one task. Do not loop. Print a one-line summary to stdout:

```
Arch-5 worker: issue #<n> moved to needs_review, branch task/<task_id>, PR #<pr_number>
```

## Failure handling

If any step fails unrecoverably:

1. Comment on the issue with the error and what was attempted.
2. Swap labels back to `status:failed`.
3. Exit non-zero so the scheduled-agent log flags the run as failed.

Do **not** try to clean up the task branch — leave it so a human can inspect.

## Things you must never do

- Touch any case other than the one named in the task.
- Close the issue (landmark detector owns closing).
- Edit between `<!-- roscoe-*-start -->` markers.
- Skip the data contract.
- Push directly to `main`. Always use `task/<task_id>` branches.
- Run more than one task per invocation.
